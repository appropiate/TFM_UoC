1 Abstract

En este informe se muestran genes diferencialmente expresados en muestras de pacientes con mieloma múltiple, así como los correspondientes procesos biológicos enrriquecidos.

2.2 Data and study design

El conjunto de datos usados en este informe procede de muestras de sangre de adulto sano y con mieloma múltiple del estudio con PMID:24816239 para detectar genes diferencialmente expresados mediante el uso de la tecnología RNAseq (plataforma Illumina Genome Analyzer II) [1]. Los datos completos originales se encuentran disponibles en el repositorio Gene expression omnibus (GEO) bajo el accession number GSE47552. No obstante, en este estudio sólo se analizan datos de pacientes sanos vs con mieloma múltiple.

En cuanto al tipo de estudio, el análisis aquí presentado puede clasificarse como un estudio comparativo (Class comparison), ya que se comparan muestras de pacientes sanos ó con mieloma múltiple.

En lo que respecta al diseño experimental, miramos los siguientes puntos:

  • Fuentes de variabilidad: Quizás la obtención de muestras pudiera haber sido algo hetereogénea entre muestras, lo que podría introducir variabilidad en los resultados, pero el hecho de tener réplicas biológicas podría minimizar su impacto. Si a partir de aquí las muestras se obtienen o procesan de manera homogénea (en paralelo o con un bloqueo adecuado), la variabilidad no debida a las diferentes condiciones se pueda minimizar. Valoraremos si las fechas de ejecución de los microarrasys dan lugar a un batch factor que pueda influenciar los resultados del análisis.

  • No se han empleado réplicas técnicas.

  • Réplicas biológicas: Cinco de pacientes sanos y 41 con mieloma múltiple.

  • Aleatorización: No procede en este caso al estar comparando.

El pipeline seguido para el análisis mostrado en este informe es análogo al usado en el informe Case_Study_1-Microarrays_Analysis.html que se puede encontrar en github:

  3. Normalización y visualización de datos normalizados.
  4. Batch detection.
  5. Detección de genes con mayor variación.

  7. Generación de matriz de diseño y matriz de contraste.
  8. Estimación del modelo para selección de genes diferencialmente expresados.
  9. Anotación de genes diferencialmente expresados.
  10. Selección y visualización de genes diferencialmente expresados: Diagramas Venn y Heatmaps.
  11. Análisis de significación biológica de los resultados.
  12. Conclusión.

library(ggplot2)
library(ggrepel)
plotPCA3 <- function (datos, labels, factor, title, scale,colores, size = 1.5, glineas = 0.25) {
  data <- prcomp(t(datos),scale=scale)
  
  # plot adjustments
  dataDf <- data.frame(data$x)
  Group <- factor
  loads <- round(data$sdev^2/sum(data$sdev^2)*100,1)
  
  # main plot
  p1 <- ggplot(dataDf,aes(x=PC1, y=PC2)) + theme_classic() +
    geom_hline(yintercept = 0, color = "gray70") +
    geom_vline(xintercept = 0, color = "gray70") +
    geom_point(aes(color = Group), alpha = 0.55, size = 3) +
    coord_cartesian(xlim = c(min(data$x[,1])-5,max(data$x[,1])+5)) +
    scale_fill_discrete(name = "Group")
  
  # avoiding labels superposition
  p1 + geom_text_repel(aes(y = PC2 + 0.25, label = labels),segment.size = 0.25, size = size) + 
    labs(x = c(paste("PC1",loads[1],"%")),y=c(paste("PC2",loads[2],"%"))) + 
    ggtitle(paste("Principal Component Analysis para",title,sep=" ")) + 
    theme(plot.title = element_text(hjust = 0.5)) +
    scale_color_manual(values=colores)
}

3.4 Batch detection

Puede ser que los microarrays de este estudio se hayan visto afectados por variables no biológicas tales como el uso de reactivos de diferentes lotes, diferentes técnicos llevando a cabo el protocolo, o la más común, la fecha de procesado. El hecho de que en el estudio no se mencione la existencia de bloqueo puede hacer pensar que las muestras se hayan procesado de manera homogénea, bien todas el mismo día, o bien con muestras de cada grupo incluídas en cada día de procesamiento de muestras. Sin embargo, a partir de la función get.celfile.dates() aplicada a los archivos .CEL podemos ver que todos los análisis con microarrays no se realizaron el mismo día, sino que se distribuyeron en diferentese fechas de manera no totalmente homogénea, lo cual quizás haya podido introducir variabilidad en los datos obtenidos y produciendo el denominado “batch effect”:

Para analizar un posible “batch effect” debido a la fecha de procesado se usará el método Principal Variation Component Analysis (PVCA) mediante la función pvcaBatchAssess() del paquete pvca. Este método estima la fuente y proporción de variación en dos pasos: el principal component analysis, y el variance component analysis. Usaremos la condición experimental “Group” como referencia para saber en que medida la fecha de procesado (“batch factor”) ha podido afectar a los datos obtenidos. Para ello creamos la variable Process.date con dichas fechas y las añadimos a targets:

3.6 Filtering of genes with lowest variability

Con el objetivo de aumentar la potencia de nuestro análisis y por tanto reducir el error tipo II (falsos negativos) conviene reducir el número de genes a los que les aplicamos el contraste para determinar si están diferencialmente expresados. Para ello eliminaremos los genes con menos probabilidad de estar expresados (es decir, aquéllos que tengan menor variabilidad) mediante el uso de la función nsFilter del paquete geneFilter. También filtraremos sondas sin un identificador del gen asociado, para lo cual debemos usar el paquete de anotación correspondiente al chip usado en este estudio (hugene10sttranscriptcluster.db):

library(genefilter)
library(hugene10sttranscriptcluster.db)
annotation(eset_rma) <- "hugene10sttranscriptcluster.db"
filtered <- nsFilter(eset_rma, 
                     require.entrez = TRUE, remove.dupEntrez = TRUE,
                     var.filter=TRUE, var.func=IQR, var.cutoff=0.75, 
                     filterByQuantile=TRUE, feature.exclude = "^AFFX")

3.10 Gene Annotation

Para anotar las sondas de los microarrays con sus correspondientes nombres de genes, símbolos e identificadores usamos la función annotatedTopTable que se muestra a continuación, la cual extrae las columnas pertinentes de nuestra base de datos de anotaciones (hugene10sttranscriptcluster.db) y las fusiona con nuestras TopTabs. Para ello, previamente se debe añadir una columna con los PROBEIDs a las TopTabs mostradas en el apartado anterior que servirá de referencia para dicha fusión:

annotatedTopTable <- function(topTab, anotPackage)
{
  topTab <- cbind(PROBEID=rownames(topTab), topTab) # Se añade columna con PROBEIDs
  myProbes <- rownames(topTab)
  
  # Objeto ChipDb a partir de paquete de anotación:
  thePackage <- eval(parse(text = anotPackage)) 
  
  # Selección de columnas de interés del objeto ChipDb:
  geneAnots <- AnnotationDbi::select(thePackage, myProbes, c("SYMBOL", "ENTREZID", "GENENAME")) 
  
  # Fusión de columnas seleccionadas en paso anterior con TopTabs según PROBEID:
  annotatedTopTab<- merge(x=geneAnots, y=topTab, by.x="PROBEID", by.y="PROBEID") 
return(annotatedTopTab)
}
